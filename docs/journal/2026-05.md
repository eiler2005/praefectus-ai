## 2026-05-09 — Critical container memory limit raised by 200 MB

**What:** the host-level memory limit for the main gateway container was raised from 1024 MB to 1224 MB.

**Why:** the container was healthy but `RestartCount=90`; the kernel journal showed cgroup OOM kills inside its Docker scope.

**How:** via `70-docker-limits-critical.yml` — persistent `docker-compose.override.local.yml` plus an immediate `docker update --memory` (no restart).

**Result:** limit persists across the next `docker compose up`; verify after with `docker inspect` and `./verify.sh`.

## 2026-05-03 — Phases 1–3: baseline + memory limits

**What:** Bootstrapped this repo + applied `mem_limit` to every container on the host.

**Why:** the VPS was showing swap usage at 1.6 / 2 GB — clear memory pressure. The OOM killer could pick any process at random.

**How:**

- Phase 1: structure, vault, disk-cleanup, verify, git init
- Phase 2: `60-docker-limits.yml` — five medium-criticality services (256–384 MB each)
- Phase 3: `70-docker-limits-critical.yml` — four critical services with immediate `docker update --memory` plus persistent override files

**Result:**

- 11 of 13 containers carry limits (two routing-side containers stayed unbounded; measured < 60 MB each, stable).
- Sum of limits: ~ 4.4 GB on a 4 GB host (overcommit acceptable; measured worst case < 3.5 GB).
- All limits applied immediately via `docker update`, plus `override.local.yml` for persistence.

**Things to verify:** `docker stats --no-stream` after any service restart.

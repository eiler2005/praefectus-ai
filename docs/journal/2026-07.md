# July 2026

## 2026-07-05 — OpenClaw Gateway resource containment

Context:

- `maxtg_bridge` stopped delivering while the shared CX23 host was offline after a hard stop.
- The previous boot had repeated OpenClaw cgroup OOM/restart evidence and a high Gateway restart count.
- Existing CPU/RAM/PID caps were not enough because `openclaw-gateway` could keep restarting forever.

Changes:

- Keep OpenClaw Gateway at `0.90 CPU`, `1224m` RAM, no swap, and `256` pids.
- Add Gateway restart budget `on-failure:5`.
- Persist Docker json log caps for OpenClaw critical services at `10m x3`.
- Align LightRAG host limits with the live OpenClaw runbook: `2304m` RAM and `2816m` memory-swap.

Validation:

- Run `ansible-playbook --syntax-check playbooks/70-docker-limits-critical.yml`.
- Run `ansible-playbook playbooks/70-docker-limits-critical.yml --check --diff` before apply.
- After apply/recreate, verify Gateway with `docker inspect openclaw-openclaw-gateway-1` for memory,
  pids, restart policy, and logging config.

## 2026-07-06 — Early resource alerts and external watchdog

Context:

- The CX23 host still showed high swap use after OpenClaw Gateway containment.
- `openclaw-gateway` reached more than 80% of its Docker memory limit and had fresh restarts.
- Kernel logs showed a new OpenClaw cgroup OOM on `2026-07-06`.

Changes:

- Lower swap WARN threshold to `60%` while keeping CRIT at `80%`.
- Alert on Docker container memory pressure at `80%` WARN and `95%` CRIT.
- Alert on recent kernel/cgroup OOM evidence in the monitor lookback window.
- Alert on RestartCount increases for high-risk containers such as `openclaw-openclaw-gateway-1`.
- Install `vps-external-watchdog.timer` so each managed VPS checks the other managed VPS hosts over
  TCP/443 every minute and sends Telegram if a host stops answering from outside.

Validation:

- Run `ansible-playbook --syntax-check playbooks/20-monitoring.yml playbooks/99-verify.yml`.
- Run `ansible-playbook playbooks/20-monitoring.yml --check --diff` before apply.
- After apply, run `./modules/monitoring/bin/run-check --test-alert` and
  `./modules/monitoring/bin/run-check --watchdog-test` on both managed hosts.
- Re-run `./verify.sh`; current swap above `60%` should report `WARN`, not `OK`.
